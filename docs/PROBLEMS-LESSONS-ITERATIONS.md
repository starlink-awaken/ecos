## 一、最大的问题: 架构设计不做实施约束检查

```
Phase 1 设计8角色委员会 — 骄傲地写了CHARTER.md
  没人检查 delegate_task 到底支持几个并发
  发现: 最多3个
  浪费: 1份文档 + 1次推演 + 1次修正

Phase 1 设计SSB贯穿所有层 — 骄傲地画了架构图
  没人检查 Phase 1 到底有几个Agent
  发现: 1个 (Hermes自己)
  浪费: 1份Schema设计 + 1次降级文档

Phase 2 加感知层 — 骄傲地写了filter_scorer
  没人验证结构评分能否挡住真实攻击
  发现: 渗透测试2分钟构造了一份100分的垃圾文档
  浪费: 后来补的content_integrity.py
```

**根因: 先画蓝图再碰现实, 而不是先摸基础设施再画蓝图。**

修复 — 实施约束检查表 (已建立, 在 `docs/policy/CONSTRAINT-CHECKLIST.md`)。但问题不在文档, 在习惯——设计前没有"检查基础设施边界"这个动作。

## 二、第二个问题: hermes config set 的 args 陷阱

```
问题:
  一条命令: hermes config set mcp_servers.kos.args '["/path/to/kos-mcp-server.py"]'
  结果: args 变成了 JSON 字符串, 不是 YAML 列表
  后果: KOS + Minerva 两个 MCP 全部连接失败
  诊断耗时: 10 分钟
  修复: 1 行手动编辑 config.yaml

为什么错:
  以为 hermes config set 理解复杂类型
  实际上它只做字符串替换
  预期行为: YAML 列表 → 实际: JSON 字符串嵌入 YAML
```

**教训: 任何工具配置后必须验证, 不要相信它会做类型推断。** 这条坑浪费了 10 分钟, 但发现了问题的根源在于 "工具的行为假设"。

## 三、第三个问题: 过滤器有效性验证滞后

```
filter_scorer 在 Phase 2 就写好了
  684 行代码, 结构化评分, 五维检查
  但从未被攻击过 — 直到 Phase 3 渗透测试
  测试2分钟就构造了一份 100 分的垃圾文档
  过滤器形同虚设

为什么:
  写了过滤器, 但没验证过滤器的工作效果
  "写了就完了" 心态
  相信结构评分能挡住一切

修复: content_integrity.py (事后补)
  应该: 写 filter_scorer 时就该构造对抗样本测它
```

**教训: 安全类功能的验收标准不是 "写完了", 而是 "被攻击了没倒"。** 没有经过对抗测试的过滤器等于没有过滤器。

## 四、第四个问题: 跨Agent依赖未提前考虑

```
Phase 1 设计依靠 Hermes 单会话
  HANDOFF 依赖于 Agent "记得写"
  事实上 Agent 经常忘
  WF-005 补了自动更新, 但这是个后期修补

Phase 2 设计多Agent协作
  以为 delegate_task 是万能方案
  没提前验证:
    - 子Agent能否用工具 (能, 但有限)
    - 子Agent能否调MCP (不能, 只有native可以)
    - Claude CLI是否需要人工确认 (需要)
  以上问题在 Phase 3 才发现

为什么:
  假设 "基础设施能做" 而不是验证 "基础设施实际能做"
  对 Hermes 平台的能力边界不清
```

**教训: 跨Agent协作的设计必须从 "基础设施实测" 开始, 不是从 "架构理论" 开始。** 一条 delegate_task 测试能节省半小时的设计失误。

## 五、第五个问题: 安全评分与功能评分的矛盾

```
Phase 1→2: 架构 61%→78% (+17%)
Phase 1→2: 安全 72%→65% (-7%)

为什么?
  加功能 → 攻击面扩大
  感知层 + SSB + Minerva 9/9 + 5 Cron = 3个新攻击面
  防御没跟上

这是个已知 Trade-off:
  要么: 安全优先, 功能慢, 安全高
  要么: 功能优先, 功能快, 安全先降后升

我们选了 B (功能优先)
  功能从 61% 到 82%
  安全从 72% 降到 65% 又回到 78%
  这不是错误, 是选择
  但没提前告知用户这个选择
```

**教训: 架构和安全的此消彼长不是事实问题, 是沟通问题。** 如果提前说 "功能优先, 安全会先降后升", 用户就知道预期。不说, 用户看到安全下降就会觉得是失败。

## 六、迭代建议

```
现在应改的:

  1. 设计前调约束检查表 (5分钟的事)
     → 防止 Phase 4 再出现"设计8角色发现3并发"
  
  2. 安全功能的验收标准改为"攻击测试通过"
     → 而不是"代码写完"
     → filter_scorer 的教训

  3. 每次架构变更前先跑一条 delegate_task 测试
     → 验证基础设施的实际能力
     → 10秒钟的事能省半小时返工

下个Phase应改的:

  4. 提前告知用户功能vs安全的trade-off
     → 不是事后解释, 是事前预期

  5. 为每项安全功能配对抗样本
     → 像 content_integrity.py 的测试模式一样
     → 写功能时就写好对应的攻击场景
```

## 七、一句话结论

**最大的问题不是技术问题, 是"假设基础设施能做某事而不验证"的习惯。** 其他四个问题都是这个习惯的具体表现。

修复习惯只需要一个动作: 设计之前跑约束检查表, 写安全功能之前构造对抗样本。5分钟能解决 80% 的后期返工。
