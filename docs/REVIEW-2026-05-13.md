## 四、问题与教训

### 4.1 失败案例分类

| ID | 类型 | 严重度 | 教训 |
|----|------|--------|------|
| FAIL-001 | 过度设计 | MED | 架构必须标注"当前Phase最小可实施方案" |
| FAIL-002 | 规则不适配 | LOW | 阈值规则发布前必须问：是否对所有任务类型成立？ |
| FAIL-003 | 过早设计 | LOW | 降级声明必须与升级路径一样清晰 |
| FAIL-004 | 测试误报 | LOW | macOS缺少Linux工具(timeout)，MCP验证需用Python subprocess |
| FAIL-005 | FTS5限制 | LOW | Agent用单关键词搜索，Hermes做多步整合 |

### 4.2 最大教训

1. **逻辑推演是最有价值的审查手段** — 3个矛盾躺在文档里1小时，推演20分钟全暴露。此后所有架构产出都应走一轮推演。

2. **`hermes config set` 的 args 陷阱** — 存成了 JSON 字符串而非 YAML 列表。2个MCP服务器因这个1行差异全部连接失败。教训：配置后立即验证，不要假设。

3. **KOS→Minerva 桥接发现** — Minerva MCP 的"部分可用"不是 bug，是有意架构分层。深入代码才能理解设计意图。教训：不要只看表面错误信息。

4. **HANDOFF 纪律** — 机制设计正确，但 Agent 执行时疏漏了更新。这是 LADS 机制从理论到实践必须跨越的鸿沟。
