# 不可逆操作规则

> 解决 L0-02 "所有行动可撤销" 与外部 API 不可逆性的矛盾
> 版本: v0.1.0
> 创建: 2026-05-13（逻辑推演后补文档）
> 所属层: L6 反馈层（宪法执行器参考）

---

## L0-02 原文

> "所有 Agent 行动必须可追溯、可审计、可撤销"

## 现实矛盾

以下操作一旦执行，无法撤回：

| 操作类型 | 具体示例 | 不可逆原因 |
|----------|----------|------------|
| 邮件发送 | himalaya send | SMTP 协议无撤回机制 |
| 公开发帖 | xurl post, Discord 消息 | 已推送至第三方平台，撤回≠取消可见 |
| Git push | git push origin main | 已推送至远程，force push 可能破坏他人工作 |
| 数据删除 | rm -rf, DROP TABLE | 无备份则永久丢失 |
| API 写操作 | POST/PUT/DELETE 外部服务 | 不可回滚（如 GitHub Issue close 后 reopen 会产生通知噪音） |
| 通知推送 | send_message | 消息已抵达用户设备 |
| 云存储写 | gsutil cp, aws s3 cp | 覆盖操作不可逆 |

---

## 分级规则

### 一级：完全可逆（直接执行）

操作完成后可通过简单反向操作恢复，无需人类确认。

**示例：**
- 创建临时文件 → 删除文件
- 打开浏览器标签 → 关闭标签
- 写入 STATE.yaml → 回滚到上一版本（git）
- MCP 只读查询（KOS搜索、Minerva研究）

### 二级：需确认可逆（快速流程即可）

操作本身可逆，但反向操作有副作用或成本。

**规则：** 走三角模式（CHAIR+EXEC+AUDIT），AUDIT 不反对即执行。

**示例：**
- 创建 GitHub Issue → 关闭 Issue（会产生通知）
- 发送 Discord 消息 → 删除消息（可能已被看到）
- 写入文件 → 覆盖（原内容丢失）
- 创建 cron job → 删除 cron job

### 三级：不可逆（必须人类确认）

操作一旦执行无法撤销或撤销代价极高。

**规则：** 必须人类明确确认后才能执行。确认信息必须包含操作描述+不可逆声明。

**示例：**
- 发送邮件（外部）
- 公开发帖（X、Discord 公开频道）
- Git push 到共享分支
- 删除远程资源（GitHub仓库、云存储桶）
- rm -rf 无备份删除
- send_message 到非测试频道

---

## 执行前确认模板（三级操作）

当 Agent 判断某个操作属于三级不可逆时，必须呈现以下格式给人类确认：

```
⚠️ 不可逆操作确认

操作类型: [具体操作描述]
不可逆原因: [为什么无法撤回]
影响范围: [谁/什么会受影响]
替代方案: [有没有更安全的方式达到同样目标]

确认?
  [Yes] 执行
  [No]  取消
```

## 宪法执行器的检查规则（反馈层）

每次行动层执行后，宪法执行器（L6）必须检查：

```
1. 操作是否在不可逆清单中？
   → 是：检查是否有对应级别的人类确认记录
        → 有：通过
        → 无：标记违规，写入 FAILURES/

2. 操作是否可能触发 L0 边界？
   → 是：立即暂停 + 人类通知

3. 操作结果是否可追溯？
   → 检查是否有日志 → 无日志则写入 FAILURES/
```

---

## 与 L0 的关系

本文档**不修改 L0-02**。L0-02 的要求通过分级机制实现：
- "可追溯" → 日志记录（所有级别适用）
- "可审计" → 确认记录 + 审计日志
- "可撤销" → 一级和二级操作确保有回滚方案；三级操作通过人类确认替代"可撤销"（人类承担不可逆的责任）
